Architecture

How it works

A layered system where product teams move fast inside a boundary the platform team defines and maintains.

1

Product team opens a PR

A CDKTF stack change in the product repo triggers the shared CI pipeline automatically.

2

Validate & plan

terraform validate runs, then terraform plan synthesizes the CDKTF output and passes it to the OPA policy check.

3

Policy gate

Conftest evaluates the plan against the full policy suite. Any violation fails the PR — no approval can override it.

4

Merge & deploy

Dev and QA deploy automatically. Stage and prod require a platform reviewer's approval in the GitHub Environment gate.

5

Azure Policy backstop

Any out-of-band change that bypasses the pipeline is caught and denied at the ARM API level.

infra.yml → reusable-workflows 
# Product team repo

  stage → platform approval
  prod  → platform approval

Ownership model

Clear separation of concerns

Product teams write constructs. The platform team owns everything beneath.

Product team

CDKTF stacks

Write constructs in any of five languages. Install the platform construct library. Open PRs — the pipeline handles everything else.

Platform team

Construct libraries · Terraform modules · Reusable workflows · OPA policies

Own the abstraction boundary. Version and publish constructs. Review staging and prod approvals. Respond to drift alerts.

Azure

Azure Policy · Management locks · OIDC service principals · State backend

Enforced at the ARM API level. CanNotDelete locks on all non-dev resources. OIDC throughout — no static credentials stored anywhere.
